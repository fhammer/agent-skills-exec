## 🎯 核心设计要点

### 1. 分层架构 (5层)

```
┌─────────────────────────────────────────────┐
│  Layer 5: 应用接入层                          │
│  (Web App / 移动端 / 小程序 / 企业系统)        │
├─────────────────────────────────────────────┤
│  Layer 4: API 网关层                          │
│  (认证鉴权 / 限流熔断 / 路由分发 / 协议转换)  │
├─────────────────────────────────────────────┤
│  Layer 3: 智能体框架核心层                     │
│  (Agent Runtime / Skill Engine / Extension Layer) │
├─────────────────────────────────────────────┤
│  Layer 2: 基础设施层                          │
│  (多租户 / 会话 / 审计 / Token / Tool / LLM适配) │
├─────────────────────────────────────────────┤
│  Layer 1: 外部服务层                          │
│  (LLM服务 / 业务API / 数据库 / 消息队列)      │
└─────────────────────────────────────────────┘
```

### 2. 核心抽象层 (6个)

| 抽象层 | 关键接口/类 | 职责 | 扩展点 |
|--------|-------------|------|--------|
| **Agent** | `Agent` 基类 | 统一调度、状态管理、插件机制 | `register_plugin()`, `add_hook()` |
| **Skill** | `Skill` 基类 | 业务逻辑执行 | `execute()`, `@skill`装饰器 |
| **Tool** | `Tool` 基类 | 底层功能调用 | `@tool`装饰器, `register_tool()` |
| **Context** | `AgentContext` | 三层上下文管理 | 读写权限控制 |
| **Memory** | `MemoryBackend` | 记忆存储/检索 | `store()`, `retrieve()`, `search()` |

### 3. 扩展机制 (4种)

| 扩展机制 | 核心价值 | 关键API/使用方式 |
|----------|----------|-----------------|
| **Plugin系统** | 完整功能扩展 | `register_plugin()`, `on_register()` |
| **Hook系统** | 关键节点介入 | `add_hook(event, handler)`, 8个Hook点 |
| **Event系统** | 异步解耦 | `@agent.on_event()`, 事件订阅/发布 |
| **配置系统** | 动态调整 | 5层配置优先级, 热更新 |

### 4. 业务系统集成 (5个层面)

| 集成层面 | 机制 | 业务系统收益 |
|----------|------|--------------|
| **适配器层** | `BusinessAPIAdapter` | 复用现有API，零改造 |
| **数据转换层** | `DataTransformer` | 自动映射业务数据 |
| **会话管理层** | `SessionManager` | 状态持久化、多轮对话 |
| **生命周期钩子** | 8个Hook点 | 自定义逻辑注入 |
| **事件订阅** | Event System | 异步响应Agent行为 |

## 📊 架构决策记录 (ADR)

### ADR-001: 为什么选择单Agent而非Multi-Agent?

**背景**: 需要决定Agent编排架构

**决策**: 采用单Agent + 多Skill编排

**理由**:
1. **Token效率**: 单Agent避免Multi-Agent通信开销，Token增长线性而非指数
2. **调试简单**: 全链路审计，问题定位容易
3. **成本可控**: 渐进式披露策略，避免重复计算
4. **适合场景**: 电商场景多为线性依赖任务流，非并行探索

**替代方案**: Multi-Agent (AutoGen/CrewAI) - 适合探索性、并行化任务

---

### ADR-002: 三层上下文 vs 单层上下文?

**背景**: 需要设计上下文管理架构

**决策**: 采用三层上下文（User Input / Working Memory / Environment）

**理由**:
1. **职责分离**: 每层有明确职责，避免数据混乱
2. **权限控制**: 不同组件对每层有不同权限（Planner只读L1，Executor可写L2）
3. **渐进披露**: 支持Token压缩策略，控制上下文大小
4. **可观测性**: 每层独立审计，便于调试

**替代方案**: 单层扁平上下文 - 简单但难以管理复杂场景

---

### ADR-003: 为什么采用抽象基类而非鸭子类型?

**背景**: 需要设计Agent/Skill/Tool的接口契约

**决策**: 采用抽象基类（ABC）+ 类型提示

**理由**:
1. **接口契约**: 明确强制子类实现必要方法
2. **IDE支持**: 更好的代码补全和类型检查
3. **文档清晰**: 抽象类本身即文档
4. **运行时检查**: 实例化时验证完整性

**替代方案**: 鸭子类型 + Protocol - 更灵活但契约不明确

## 🚀 Phase 2 建议技术选型

### 技术栈建议

| 组件 | 推荐选型 | 备选方案 | 理由 |
|------|----------|----------|------|
| **Web框架** | FastAPI | Flask | 原生异步、自动生成API文档、类型提示支持 |
| **数据库** | PostgreSQL | MySQL | JSON支持好、事务强、生态丰富 |
| **缓存** | Redis | Memcached | 支持数据结构、持久化、发布订阅 |
| **消息队列** | Redis/RabbitMQ | Kafka | 简单场景Redis够用，复杂场景RabbitMQ |
| **部署** | Docker + Docker Compose | K8s (后期) | 开发简单，生产可迁移到K8s |
| **监控** | Prometheus + Grafana | ELK | 云原生标准，社区活跃 |
