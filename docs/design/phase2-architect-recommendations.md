| Skill自动发现 | ✅ 完成 | utils/skill_loader.py | 本地目录扫描 |

### 1.2 架构优势

1. **Token效率高** - 单Agent避免Multi-Agent通信开销，Token增长线性
2. **调试简单** - 全链路审计，问题定位容易
3. **扩展性好** - Skill即插即用，Provider抽象完整
4. **成本可控** - 渐进式披露策略，避免重复计算

### 1.3 当前限制

| 限制 | 影响 | Phase 2 需求 |
|------|------|--------------|
| 单机运行 | 无法服务多租户 | 需要API服务化 |
| 本地Skills | 无法远程加载 | 需要Skill市场 |
| 内存会话 | 无状态持久化 | 需要会话管理 |
| 同步执行 | 高并发阻塞 | 需要异步支持 |

## 三、架构设计原则

### 3.1 原则1: 渐进式复杂度

```
Phase 2 目标: 从单机本地工具 → 企业级API服务

实现路径:
  Week 1-2: 单租户API服务
    └─ FastAPI基础服务
    └─ 基础认证鉴权
    └─ 数据源连接器

  Week 3: 多租户隔离
    └─ TenantContext
    └─ 租户级配置管理
    └─ 租户隔离验证

  Week 4: 监控和运维
    └─ Metrics采集
    └─ 审计日志增强
    └─ 健康检查
```

### 3.2 原则2: 零侵入集成

```
业务系统接入成本 = 0 代码改造

实现方式:
  1. 通过适配器层对接现有API
  2. 通过配置文件映射数据格式
  3. 通过Hook点注入业务逻辑
```

### 3.3 原则3: 可观测性优先

```
所有外部调用可追溯:
  - API请求 → 审计日志
  - Skill执行 → Metrics记录
  - Token消耗 → 实时统计
```

## 四、扩展机制设计

### 4.1 五层集成设计

```
┌─────────────────────────────────────────────────────────┐
│ 业务系统                                                │
└────────────┬────────────────────────────────────────────┘
             │
┌────────────▼────────────────────────────────────────────┐
│ Layer 1: 适配器层 (BusinessAPIAdapter)                  │
│ - 将业务API适配为Tool                                   │
│ - 零代码改造，配置驱动                                   │
└────────────┬────────────────────────────────────────────┘
             │
┌────────────▼────────────────────────────────────────────┐
│ Layer 2: 数据转换层 (DataTransformer)                   │
│ - 自动映射业务数据到框架格式                             │
│ - Schema驱动的转换规则                                   │
└────────────┬────────────────────────────────────────────┘
             │
┌────────────▼────────────────────────────────────────────┐
│ Layer 3: 数据源连接器 (Connector)                       │
│ - 标准化数据库/API连接                                   │
│ - 连接池管理                                            │
└────────────┬────────────────────────────────────────────┘
             │
┌────────────▼────────────────────────────────────────────┐
│ Layer 4: Hook系统 (8个Hook点)                           │
│ - before_planning, after_skill_execute等                │
│ - 业务逻辑注入点                                         │
└────────────┬────────────────────────────────────────────┘
             │
┌────────────▼────────────────────────────────────────────┐
│ Layer 5: Agent核心框架                                   │
│ - Coordinator + Skills                                   │
└─────────────────────────────────────────────────────────┘
```

### 4.2 适配器层示例

```yaml
# 业务系统只需提供配置
business_integration:
  name: "电商系统"
  apis:
    - name: "订单查询API"
      endpoint: "https://api.example.com/orders/{id}"
      method: GET
      auth:
        type: "bearer"
        token: "${ORDER_API_TOKEN}"
      mapping:
        request:
          order_id: "$.params.id"
        response:
          order_status: "$.data.status"
          items: "$.data.items"
```

框架自动：
1. 将此API注册为Tool
2. 自动处理数据转换
3. 在Skill中可直接调用

## 六、需要确认的关键决策

### 6.1 MVP边界确认

| 问题 | 选项A | 选项B | 建议 |
|------|-------|-------|------|
| 多租户是否Phase 2必须？ | 必须，支持多客户 | 可选，先单租户 | 选项B（渐进式） |
| 异步执行是否必须？ | 必须，高并发 | 可选，同步API | 选项A（现代API） |
| 会话持久化？ | 必须，数据库 | 可选，内存 | 选项A（用户体验） |
| Webhook机制？ | 必须，事件驱动 | 可选，Phase 3 | 选项B（非核心） |

### 6.2 部署方式

| 问题 | 选项 | 说明 |
|------|------|------|
| 目标部署环境 | Docker | 推荐，标准化 |
| 是否K8s | Phase 3 | 过早优化 |
| 私有化部署 | 支持 | 企业客户需求 |

### 6.3 认证复杂度

| 阶段 | 认证方式 | 说明 |
|------|----------|------|
| MVP | API Key | 简单够用 |
| P1 | JWT Token | 标准做法 |
| P2 | OAuth2 | 企业集成 |

### 6.4 数据持久化

| 数据类型 | MVP | P1 | 说明 |
|----------|-----|----|----|
| 会话数据 | 内存 | Redis | 用户体验 |
| 审计日志 | 文件 | 数据库 | 合规需求 |
| 配置数据 | YAML | 数据库 | 热更新 |

## 七、技术栈建议

| 组件 | 推荐选型 | 备选方案 | 理由 |
|------|----------|----------|------|
| Web框架 | FastAPI | Flask | 原生异步、自动文档 |
